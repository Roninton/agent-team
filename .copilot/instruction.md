# Copilot 开发约束元规则（Meta Instruction）

## 🔴 最高优先级强制规则
1. **每次代码修改后必须先执行TypeScript错误检查，确保没有任何TS错误才能继续后续操作**
   ```bash
   # 前端TS检查
   cd src/client && npx tsc --noEmit
   
   # 后端TS检查
   cd src/server && npx tsc --noEmit
   ```
2. 任何包含TS错误的代码都不允许提交到代码仓库
3. 单元测试执行前必须先通过TS检查，否则测试结果无效

## 核心开发原则
本项目采用 **分层开发模式**，根据需求场景灵活选择开发方式，最终所有功能都必须沉淀为完整的规范文档：
1. 复杂核心功能采用 SDD(规范驱动) → TDD(测试驱动) → 代码实现 的完整流程
2. 小需求/快速迭代采用 PDD(提示词驱动) → 验证 → 沉淀SDD 的轻量化流程
3. 所有代码必须满足对应模式的规范要求并且通过测试

---

## 开发模式适配规则
我们支持三种分层开发模式，根据需求场景灵活选择：

### 1. 完整SDD模式（默认）
- 适用场景：核心功能、架构调整、复杂业务模块开发
- 要求：完整的规范文档、评审通过后再进入开发流程

### 2. PDD（Prompt-Driven Development 提示词驱动开发）模式
- **定位**：SDD弱化版，轻量化敏捷开发模式
- **适用场景**：小功能迭代、bug修复、新功能快速探索验证
- **开发流程**：
  1. 仅需清晰的prompt需求描述，不需要完整SDD文档
  2. 快速实现并验证效果
  3. 迭代2-3个版本稳定后，沉淀收敛为正式SDD规范

### 3. SDD中间态规则
- 允许SDD文档处于未完成状态，分阶段迭代完善
- 未完成的规范需要明确标记"WIP"状态和当前版本
- 核心需求明确即可进入开发，细节可以边开发边补充
- 功能上线前必须完成对应SDD文档的最终版

### 模式切换规则
- 小需求用PDD，复杂需求用SDD
- PDD模式开发的功能稳定后必须升级为SDD规范
- 任何模式下最终都需要有对应的文档沉淀

## SDD文档编写约束
### 1. 文档质量要求
- 所有规范文档必须清晰、无歧义，具备可执行性
- 接口定义必须包含完整的参数、返回值、错误码说明
- 交互流程必须包含完整的时序、状态变化说明
- 必须包含明确的验收标准，可直接用于测试用例编写

### 2. 文档格式要求
- 统一使用Markdown格式，文件名采用小写+短横线格式
- 文档层级清晰，标题层级不超过4级
- 代码块必须指定语言类型
- 图表优先使用Mermaid语法，便于版本控制

### 3. 文档索引与引用规则
- 所有新增的SDD文档必须第一时间加入`docs/SPEC_INDEX.md`索引，禁止存在未被索引的规范文档
- 文档之间引用必须使用相对路径，禁止绝对路径引用
- 严格禁止内容冗余，相同的定义只能存在于最顶层的文档中，下层文档只能引用，不能重复定义
- 文档更新时必须同步更新索引中的说明，确保索引和实际文档内容一致

### 4. 文档评审规则
- 所有新增/修改的规范文档必须经过用户确认才能生效
- 文档变更必须记录变更原因、变更内容、变更时间
- 涉及架构调整的文档需要单独评审

---

## 上下文获取与任务创建规范（最高优先级）
### 🔴 核心原则：大改动必须先建Task，先收集context
当改动涉及以下情况时，**绝对禁止直接开始写代码**：
1. 涉及多个文件的重构或功能开发
2. 需要查阅多个SDD规范文档
3. 预计工时超过2小时
4. 影响多个模块或子系统

---

### 大改动开发三部曲（强制执行）

#### 第一步：建立Task文件（P0优先级）
- 在`task/todo/`目录创建Task文件
- 填写任务描述、验收标准、优先级
- 明确关联的SDD规范文档路径
- **未创建Task前禁止任何代码修改**

#### 第二步：上下文收集与摘要（最关键，防止上下文过载）
**严格按以下流程执行，避免上下文过载：**

1. **读索引，定范围**
   - 先读`docs/SPEC_INDEX.md`
   - 只标记当前任务**真正相关**的文档路径
   - 禁止无差别读取所有SDD文档

2. **分层读取，按需摘录**
   - Level 0（必读）：本文件 + DEVELOPMENT_GUIDELINE.md + tech-stack-and-architecture.md
   - Level 1（按需）：只读取相关子系统的架构规范
   - Level 2-4（按需）：只读取直接相关的功能/协议/UI规范

3. **摘要写入Task（核心机制）**
   - 将所有**核心约束、设计原则、接口定义、命名规范**摘录到Task的「上下文信息」章节
   - **只保留与当前任务直接相关的内容**，无关内容不摘录
   - 标注每个摘要对应的文档路径，方便回溯
   - ✅ **优势**：开发时只需读取Task文件，无需反复读取所有SDD，避免上下文过载

4. **完整性检查**
   - 确保开发前对需求和约束有100%完整理解
   - 上下文不完整禁止进入开发阶段

#### 第三步：逐步执行并更新Todo
1. 将任务拆解为具体执行步骤（分阶段）
2. 每完成一个步骤立即在Task中标记✅
3. 遇到问题或变更及时更新Task记录
4. 全部完成后进行最终验证

---

### 上下文获取通用规则（所有任务适用）
1. **按索引顺序读取**：SPEC_INDEX.md → Level 0 → Level 1 → Level 2-4
2. **按需读取**：不要读取与当前任务无关的文档，避免信息噪音
3. **摘录到Task**：所有关键信息必须写入Task，开发时只依赖Task中的上下文摘要
4. **疑问回溯**：对规范有疑问时，回到原文档确认，禁止凭记忆或脑补开发

## 核心执行规则（精简版）

### 1. 任务执行
- 同一时间只能处理一个in-progress状态的任务
- 开发依赖**Task文件中的上下文摘要**，不再反复读取SDD
- 发现规范问题必须先更新文档，不能自行修改

### 2. 代码约束
- 遵循现有代码风格和架构模式
- 新增功能必须配套单元测试
- 所有对外接口必须有完整TypeScript类型定义
- 必须处理所有错误场景，禁止静默失败

### 3. 交流约束
- 技术选型、架构调整必须先确认
- 不确定的需求必须先澄清，不能假设
- 关键决策必须记录到对应文档

---

## SDD → TDD 流程
1. SDD规范完成 → 2. 提取测试点 → 3. 编写测试用例 → 4. 实现功能 → 5. 回归验证

---

## 🔧 问题修复验证流程（强制执行）
所有问题修复必须严格遵循以下三步验证流程，确保修复质量和代码稳定性：
1. ✅ 单元测试全量通过：修复完成后先运行对应模块的所有单元测试，确保全部通过，没有新增失败用例
2. ✅ E2E用例重试验证：重新执行对应的E2E测试用例，模拟真实用户场景验证问题确实已修复
3. ✅ 补充缺失测试用例：对于单元测试未覆盖到的问题场景，必须补充对应的UT用例，避免后续回归出现相同问题

---

## 工具调用核心规则（防中断）
1. **前置检查**：创建文件前检查是否已存在，编辑前先读取上下文
2. **参数校验**：使用绝对路径，替换字符串确保唯一匹配
3. **内容控制**：单次操作不超过5个替换/1000行，长内容分块处理
4. **异常重试**：失败自动重试2次，连续失败及时反馈
5. **状态跟踪**：多步骤任务记录进度，中断后从断点续接

---

## 🟡 迭代优化提示（低优先级，工作流持续改进）

### 1. SDD 对齐原则
- **合理实现优先于 SDD 文档**
  - 如果实际代码实现比 SDD 设计更合理（架构更优、职责更清晰、复杂度更低）
  - ✅ 优先保留合理实现
  - ⚠️ **必须通知用户，确认是否同步更新 SDD 文档**

### 2. 版本路标最佳实践
SDD 文档建议采用三阶段路标，避免过度设计：
```
v1 = 已实现（必须与代码 100% 一致）
v2 = 规划中（近期要做，有明确需求场景）
v3 = 展望中（未来可能，不承诺）
```

### 3. 过度设计快速自检三问法
设计复杂功能前先问自己：
1. 是核心定位必需的吗？
2. 没有它用户能用吗？
3. 未来 3 个月内会用到吗？

只要有一个"否"，就考虑降级或延后。

### 4. 测试排查通用核心原则（低优先级，适用于所有测试场景）
1. **事实优先，杜绝猜测**：所有问题先找实际证据（测试截图、日志、接口返回、手动页面验证），不确认实际情况绝不修改代码
2. **分层排查**：从环境连通性→后端接口→前端页面→测试逻辑逐层排查，优先验证基础环境问题（端口、服务是否运行、配置是否正确）
3. **最小化验证**：遇到问题先拆解最小流程验证，去掉非必要步骤只跑核心流程，快速定位问题层级

> **🔴 重要**：修改 SDD 文档架构定位、删除/新增核心功能必须先通知用户，确认后再执行。
