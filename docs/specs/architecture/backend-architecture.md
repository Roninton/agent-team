# 后端服务整体架构 SDD 规范

## 1. 概述
本文档定义后端服务的整体架构设计，采用NestJS作为核心框架，基于DDD领域驱动设计思想，分层解耦，模块化设计，支持高可用、高性能、易扩展，为整个多AI协作平台提供稳定可靠的后端服务支撑。

## 2. 设计原则
1. **分层架构**：清晰的分层设计，各层职责单一，依赖关系明确
2. **模块化**：业务功能模块化，模块之间低耦合，高内聚，可独立开发、测试、部署
3. **领域驱动**：按照业务领域划分模块，核心领域逻辑独立，不受外部依赖影响
4. **可扩展**：接口化设计，支持插件扩展，方便新增功能和对接第三方系统
5. **高可用**：无状态设计，支持水平扩展，关键组件冗余，故障自动转移
6. **高性能**：缓存、异步、并行等优化手段，保证低延迟、高吞吐量
7. **可观测**：完善的监控、日志、链路追踪，方便问题排查和性能优化
8. **安全性**：身份认证、权限控制、数据加密、防攻击等安全机制完备

### 3.1 存储架构说明
采用混合存储方案，避免不必要的重型组件依赖：
1. **Agent配置文件存储**：
   - Agent元数据配置（名称、启动命令、参数、限流配置等）以JSON格式存储在文件系统中
   - 存储路径规则：`{数据根目录}/agents/{agentId}.json`，每个代理一个独立配置文件
   - 默认数据根目录：项目根目录下的`.teamagents/data`，测试环境为`.teamagents/data-test`
   - 系统启动时会自动扫描该目录下所有JSON配置文件，自动加载并启动所有已配置的Agent
   - 支持用户直接编辑配置文件，重启后自动生效
   
2. **Agent运行时交互说明**：
   - Agent添加：前端提交创建请求后，后端会先将配置持久化到磁盘，再启动Agent进程
   - Agent删除：删除请求会同时停止Agent进程并删除磁盘上的配置文件
   - 数据隔离：不同环境（生产/测试）的代理配置完全隔离，存储在不同的目录下

2. **运行时缓存存储**：
   - 会话状态、在线状态、临时消息、任务进度等实时数据存储在Redis中
   - 支持快速读写，过期自动清理
   - 可选开启持久化，防止重启丢失数据

3. **历史数据存储**：
   - 消息历史、任务记录、操作日志等结构化数据默认存储在SQLite中
   - 单文件存储，无需额外数据库服务，部署简单
   - 高并发场景下可切换为PostgreSQL，无需修改业务代码

## 4. 分层架构
采用经典的四层分层架构，从上到下依次为：
```
┌─────────────────────────────────────────┐
│ 接入层 (API Layer)                       │
│  - Controller、WebSocket Gateway         │
│  - 处理请求、参数校验、响应格式化        │
└─────────────────────────────────────────┘
              ↓ 依赖
┌─────────────────────────────────────────┐
│ 应用层 (Application Layer)               │
│  - Service、Command、Query               │
│  - 业务流程编排、事务控制、权限校验      │
└─────────────────────────────────────────┘
              ↓ 依赖
┌─────────────────────────────────────────┐
│ 领域层 (Domain Layer)                    │
│  - Entity、Value Object、Domain Service  │
│  - 核心业务逻辑、领域规则、不变性约束    │
└─────────────────────────────────────────┘
              ↓ 依赖
┌─────────────────────────────────────────┐
│ 基础设施层 (Infrastructure Layer)        │
│  - Repository、External Service Client   │
│  - 数据库访问、缓存、消息队列、第三方调用│
└─────────────────────────────────────────┘
```

### 4.1 分层职责说明
#### 接入层 (API Layer)
- 职责：处理外部请求，提供HTTP API和WebSocket接口
- 包含组件：
  - Controller：处理HTTP请求，路由映射，参数校验，调用应用层服务，返回响应
  - WebSocket Gateway：处理WebSocket连接，消息收发，事件路由
  - DTO（Data Transfer Object）：定义请求和响应的数据结构，参数校验规则
  - 过滤器（Filter）：全局异常处理，响应格式化，错误码统一
  - 守卫（Guard）：身份认证，权限校验，接口访问控制
  - 拦截器（Interceptor）：请求日志，响应转换，性能监控，缓存控制

#### 应用层 (Application Layer)
- 职责：业务流程编排，协调各个领域服务完成业务需求
- 包含组件：
  - Service：应用服务，实现业务用例，编排领域服务和基础设施服务
  - Command：写操作命令，CQS模式，处理数据修改操作
  - Query：读操作查询，CQS模式，处理数据查询操作
  - DTO：应用层内部使用的数据传输对象
  - 事务管理：控制业务事务的边界，保证数据一致性

## 5. 目录结构与数据存储规范

### 5.1 整体目录结构原则
所有运行时数据、缓存、日志与源代码完全分离，统一存放在项目根目录下的`.teamagents`目录中，避免污染代码仓库，便于清理、迁移和备份。

### 5.2 数据目录规范
```
项目根目录/
├── .teamagents/                          # 统一的运行时数据存储目录（自动生成，可随时删除）
│   ├── data/                             # 生产环境数据根目录
│   │   ├── db/                           # SQLite数据库文件存储目录
│   │   │   └── acp.sqlite                # 主数据库文件，存储消息、任务、会话等结构化数据
│   │   ├── logs/                         # 日志文件存储目录
│   │   │   ├── app.log                   # 应用主日志
│   │   │   ├── error.log                 # 错误日志
│   │   │   └── access.log                # 访问日志
│   │   └── agents/                       # Agent配置文件存储目录
│   │       ├── {agentId}/                # 单个Agent的配置目录
│   │       │   ├── agent.md              # Agent基础配置
│   │       │   └── instruction.md        # Agent指令配置
│   │       └── ...
│   └── data-test/                        # 测试环境数据根目录（E2E测试专用，与生产环境完全隔离）
│       └── 结构同data/                   # 测试数据目录结构和生产环境完全一致
```

### 5.3 路径配置规则
1. **默认路径**：无需额外配置，默认自动在项目根目录生成`.teamagents`目录
2. **自定义路径**：可通过环境变量`ACP_DATA_ROOT`指定自定义数据根目录，如：
   ```bash
   ACP_DATA_ROOT=/path/to/custom/data npm run dev
   ```
3. **测试路径**：E2E测试自动使用`.teamagents/data-test`目录，不会污染生产数据

### 5.4 管理规则
1. `.teamagents`目录属于运行时缓存目录，**不纳入Git版本控制**，已默认添加到`.gitignore`
2. 可以随时删除整个`.teamagents`目录，重启服务会自动重建所需的目录结构
3. 测试结束后可直接删除`.teamagents/data-test`目录清理测试数据
4. 生产环境备份只需备份整个`.teamagents`目录即可完整备份所有数据
  - 权限控制：业务层面的权限校验，判断用户是否有权限执行操作

#### 领域层 (Domain Layer)
- 职责：核心业务逻辑实现，封装领域规则，是系统的核心，不依赖任何外部框架
- 包含组件：
  - Entity：领域实体，包含属性和业务方法，封装领域逻辑
  - Value Object：值对象，描述领域的属性，不可变，相等性通过属性判断
  - Aggregate：聚合根，一组相关实体和值对象的根，保证数据一致性
  - Domain Service：领域服务，封装跨实体的业务逻辑
  - Repository Interface：仓储接口，定义领域层需要的数据访问能力
  - Domain Event：领域事件，实现领域内的解耦通信
  - 领域规则：核心业务规则、不变性约束，保证数据的正确性

#### 基础设施层 (Infrastructure Layer)
- 职责：为上层提供技术支撑，实现领域层定义的接口，对接外部资源
- 包含组件：
  - Repository Implementation：仓储实现，实现领域层定义的仓储接口，访问数据库
  - External Service Client：外部服务客户端，调用第三方API、ACP代理等
  - Cache Service：缓存服务，实现Redis缓存操作
  - Message Queue Producer/Consumer：消息队列生产者和消费者，实现异步任务处理
  - 配置管理：系统配置的读取和管理
  - 工具类：通用的技术工具，加密、序列化、文件操作等

### 4.2 依赖规则
- 上层依赖下层，下层不依赖上层
- 依赖关系只能是从外到内：接入层 → 应用层 → 领域层 ← 基础设施层
- 领域层是核心，不依赖任何其他层，其他层依赖领域层
- 基础设施层实现领域层定义的接口，依赖注入到上层使用
- 层之间通过接口交互，实现解耦，方便替换实现

## 5. 模块化设计
按照业务领域划分为多个独立模块，每个模块包含完整的四层结构，模块之间通过公开的API交互，不直接依赖内部实现。

### 5.2 模块依赖规则
1. 业务模块可以依赖核心模块和支撑模块，不能反向依赖
2. 支撑模块之间可以互相依赖，但避免循环依赖
3. 业务模块之间只能通过公开的API接口交互，不能直接依赖其他模块的内部实现
4. 每个模块有独立的版本号，升级时保证向下兼容
5. 模块可以独立开启/关闭，不影响其他模块的运行

## 6. 核心架构特性
### 6.1 统一API响应格式
所有HTTP接口返回统一的响应格式，方便前端处理：
```typescript
interface ApiResponse<T = any> {
  code: number;          // 响应码，0表示成功，非0表示失败
  message: string;       // 响应消息，成功为"success"，失败为错误信息
  data: T;               // 响应数据
  requestId: string;     // 请求ID，用于链路追踪
  timestamp: number;     // 响应时间戳
}
```
错误码规范：
- 0：成功
- 1xxxx：通用错误（参数错误、认证失败、权限不足等）
- 2xxxx：业务错误（群组不存在、任务不存在、代理离线等）
- 3xxxx：系统错误（数据库错误、缓存错误、第三方调用失败等）
- 4xxxx：第三方服务错误

### 6.2 全局异常处理
所有异常统一处理，自动转换为标准API响应，记录错误日志，包含请求ID、错误栈、上下文信息，方便问题排查。

### 6.3 统一认证机制
- 采用JWT Token认证，Token有效期2小时，Refresh Token有效期7天
- 支持多端登录，每个端有独立的Token，可以单独登出
- 支持API Key认证，方便第三方系统对接
- 接口权限控制：基于角色的权限控制（RBAC），支持细粒度的接口权限配置

### 6.4 事务管理
- 使用数据库事务保证数据一致性，复杂业务操作使用分布式事务
- 事务边界在应用层控制，领域层不处理事务
- 支持事务嵌套、事务回滚、事务监听器

### 6.5 缓存策略
- 多级缓存：本地缓存（LRU） + 分布式缓存（Redis）
- 缓存失效策略：主动更新 + 超时过期，避免缓存雪崩、缓存击穿、缓存穿透
- 热点数据永久缓存，普通数据根据访问频率设置过期时间
- 缓存更新保证最终一致性，通过消息队列异步更新缓存

### 6.6 异步处理
- 所有非核心路径的操作异步处理，提升响应速度
- 使用消息队列实现异步任务，支持重试、死信、延迟调度
- 异步任务状态可追踪，失败告警，支持手动重试

### 6.7 可观测性
- 全链路追踪：每个请求生成唯一Request ID，贯穿整个调用链
- 结构化日志：所有日志包含Request ID、用户ID、模块、耗时等上下文信息
- 指标采集：采集QPS、延迟、错误率、数据库查询耗时、缓存命中率等核心指标
- 健康检查：提供HTTP健康检查接口，支持Kubernetes等容器编排系统的健康检测

## 10. 验收标准
1. 项目可以正常启动，所有模块加载成功，无依赖错误
2. 接口响应符合统一格式，异常处理正常，错误码规范
3. 认证授权功能正常，未登录用户无法访问需要权限的接口
4. 数据库操作正常，事务回滚正确，数据一致性保证
5. 缓存功能正常，命中率>90%，缓存更新及时无脏数据
6. 消息队列功能正常，异步任务可以正常执行，失败自动重试
7. 水平扩展正常，新增实例可以正常接入负载均衡，流量均匀分配
8. 监控指标正常，可以采集到QPS、延迟、错误率等核心指标
9. 日志记录完整，包含请求ID、用户ID、耗时等上下文信息，方便排查问题
10. 单元测试覆盖率>80%，集成测试覆盖率>70%，核心功能测试覆盖100%
