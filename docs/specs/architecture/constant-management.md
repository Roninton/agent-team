# 常量管理规范

## 1. 目的与范围
本文档定义项目中常量的统一管理规范，确保全栈代码、测试代码的一致性，避免硬编码导致的维护问题，符合SDD→TDD的开发流程要求。

适用范围：所有前后端TypeScript代码、测试相关代码。

## 2. 设计方案

### 2.1 常量分层设计
采用三层常量结构，按职责划分：

| 层级 | 存放位置 | 职责 | 访问范围 |
|------|----------|------|----------|
| 共享层 | `src/shared/constants/` | 前后端通用常量（UI文本、API路径、Agent类型、业务枚举等） | 前端、后端、测试代码均可引用 |
| 前端层 | `src/client/src/constants/` | 前端专用常量（路由路径、组件默认配置、样式常量等） | 仅前端代码引用 |
| 后端层 | `src/server/src/constants/` | 后端专用常量（数据库表名、缓存键、业务配置默认值等） | 仅后端代码引用 |

### 2.2 常量命名规范
1. 常量文件命名：`[领域].constants.ts`，例如：`ui.constants.ts`、`agent.constants.ts`
2. 常量对象命名：采用大写+下划线格式，使用`as const`确保类型安全
3. 常量属性命名：采用大写+下划线格式，语义明确，避免缩写
4. 枚举类型常量使用`enum`定义，普通常量使用`const object`定义

### 2.3 引用规则（强制执行）
1. **禁止硬编码**：所有文本、枚举值、路径、配置项必须引用常量，不允许直接写字面量
2. **优先使用共享常量**：前后端通用的常量必须放在`shared/constants`中，避免重复定义
3. **测试与生产代码对齐**：所有测试相关代码必须直接引用生产代码的共享常量，不允许单独定义相同值的常量
4. **常量不可修改**：常量仅用于定义固定值，禁止在运行时修改常量内容

### 2.4 类型安全要求
所有常量必须使用`as const`声明，确保TypeScript可以推导出精确的字面量类型：
```typescript
// 正确示例 ✅
export const BUTTON_TEXTS = {
  ADD_AGENT: '添加代理',
  SAVE: '确 定'
} as const; // 确保类型为字面量类型，而不是普通string

// 错误示例 ❌
export const BUTTON_TEXTS = {
  ADD_AGENT: '添加代理',
  SAVE: '确 定'
}; // 类型被推导为 { [key: string]: string }，丢失类型安全
```

## 3. 验收标准
1. 项目中无硬编码的文本、路径、枚举值，所有字面量均来自常量引用
2. 前后端、测试代码引用同一套共享常量，值完全一致
3. 所有常量文件均符合命名规范和类型安全要求
4. 修改常量值时，所有引用位置自动同步更新，无需逐一修改
5. 无重复定义的常量，相同含义的常量在项目中仅定义一次
