# 技术栈与整体架构规范

## 1. 概述
本文档定义项目的整体技术选型、分层架构设计、目录结构规范，是所有开发工作的基础架构约束。

## 2. 技术选型决策
| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| 前端 | React 18 + TypeScript + TailwindCSS | 1. 原项目已使用React，技术栈延续 <br> 2. 支持同时嵌入VS Code Webview和独立浏览器运行 <br> 3. 开发效率高，生态完善 |
| 后端 | Node.js + NestJS + TypeScript | 1. 全栈TS技术统一，降低跨语言开发成本 <br> 2. 异步非阻塞IO模型完美适配多AI并行调度的IO密集型场景 <br> 3. 可以直接复用ACP协议官方Node.js SDK，对接成本为0 <br> 4. NestJS是企业级Node.js框架，架构规范，扩展性强 |
| 数据存储 | SQLite + TypeORM | 1. 轻量级，零配置，开箱即用 <br> 2. 足够支撑单实例万级以下的数据量需求 <br> 3. 后续可以无缝迁移到PostgreSQL等更高级数据库 |
| 通信协议 | ACP协议 + WebSocket | 1. 基于官方ACP协议实现AI代理通信 <br> 2. 前后端、多端之间使用WebSocket实现实时消息推送 |

## 3. 整体分层架构
采用三层解耦架构，各层职责单一，独立部署：
```
┌─────────────────┐    ┌─────────────────┐
│   前端展示层     │    │ VS Code 扩展层  │
│ (React 双端)    │    │  (编辑器集成)   │
└─────────────────┘    └─────────────────┘
          │                     │
          └─────────┬───────────┘
                    ▼
┌─────────────────────────────────────┐
│           后端服务层                 │
│  (NestJS + Node.js)                 │
│  ├─ 群组管理模块                     │
│  ├─ 代理调度模块                     │
│  ├─ 任务分配模块                     │
│  ├─ 上下文同步模块                   │
│  ├─ 消息路由模块                     │
│  └─ ACP协议适配层                    │
└─────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────┐
│          ACP代理进程层               │
│  (每个代理独立子进程，完全隔离)      │
│  ├─ AI主管代理                       │
│  ├─ Worker代理1                      │
│  ├─ Worker代理2                      │
│  └─ ...                              │
└─────────────────────────────────────┘
```

## 4. 各层详细说明
### 4.1 前端展示层
- 支持两种运行模式：
  1. 嵌入式：作为VS Code Webview嵌入编辑器中，深度集成编辑器能力
  2. 独立式：运行在浏览器中，支持跨设备访问，不依赖VS Code
- 核心功能：
  - 多代理群聊界面
  - 群组管理、成员配置
  - 任务看板、进度展示
  - 上下文管理、配置中心

### 4.2 后端服务层
- 核心职责：
  - 管理所有AI代理的生命周期
  - 实现群组逻辑、任务分配、上下文同步
  - 消息路由，实现多端消息同步
  - 权限控制、数据持久化
  - ACP协议的统一适配和管理
- 部署模式：支持单实例部署，后续可扩展为分布式集群

### 4.3 ACP代理进程层
- 每个AI代理都是独立的Node.js子进程，完全隔离
- 每个代理有自己独立的上下文、配置、运行环境
- 与后端通过标准ACP协议通信，不需要修改代理本身的代码
- 支持所有兼容ACP协议的AI代理无缝接入

## 5. 项目目录结构规范

### 5.1 整体目录结构

```
<root>
├── .copilot/               # Copilot元约束文档
├── config/                  # 🆕 配置文件目录（独立可挂载）
│   ├── config.example.yml  # 配置模板（提交）
│   └── .env.example       # 环境变量模板（提交）
│   # config.yml       # 用户实际配置（不提交）
├── data/                    # 🆕 运行时数据目录（自动创建，不提交）
│   ├── db/                 # SQLite数据库文件
│   ├── logs/               # 日志文件
│   └── agents/             # 代理工作目录
├── docs/                   # SDD规范文档
│   ├── specs/              # 规范文档
│   ├── api/                # API文档
│   └── user/               # 用户文档
├── src/                    # 源代码（只读，运行时不写入）
│   ├── client/             # 前端React应用
│   ├── server/             # 后端NestJS应用
│   └── shared/             # 前后端共享代码
├── task/                   # LFT任务跟踪目录
```

### 5.2 目录设计原则

| 目录 | 职责 | Git 状态 | 备份需求 |
|------|------|----------|---------|
| `config/` | 用户配置文件 | 模板提交，用户配置不提交 | 需要 |
| `data/` | 运行时生成数据 | 全部忽略，不提交 | **必须备份** |
| `src/` | 源代码 | 完整提交 | 代码仓库本身就是备份 |

## 6. 配置系统架构规范

### 6.1 三级配置优先级

```
优先级从高到低：
1. 环境变量 (ACP_* 前缀）
2. YAML 配置文件 (config.yml)
3. 代码默认值 (config.default.ts)
```

### 6.2 配置文件搜索顺序

按优先级从高到低搜索：
1. 环境变量 `ACP_CONFIG_FILE` 指定路径
2. 系统级 `/etc/acp-platform/config.yml` (Linux)
3. 用户级 `~/.config/acp-platform/config.yml`
4. 项目级 `./config/config.yml` ✅ 推荐
5. 以上都找不到 → 使用代码默认值

### 6.3 ConfigService 架构位置

```
后端服务层
├── ConfigService (核心配置模块，最先初始化）
│   ├── 配置加载 (YAML + 环境变量合并）
│   ├── 配置验证（类型、范围校验）
│   └── 目录自动创建
│   └── 统一配置读取入口
├── AgentService / SessionService / ...（其他模块依赖 ConfigService）
```

### 6.4 配置开发规范

✅ **所有模块必须通过 ConfigService 读取配置**
```typescript
// ✅ 正确方式
constructor(private configService: ConfigService) {
  const port = this.configService.get('server.port');
}

// ❌ 禁止方式
process.env.ACP_SERVER_PORT  // 禁止直接读取环境变量！
```

## 8. API 接口更新
├── src/                    # 所有源代码统一放在src下
│   ├── client/             # 前端React项目
│   ├── server/             # 后端NestJS项目
│   ├── shared/             # 前后端共享的TypeScript类型、工具函数
│   └── extension/          # VS Code扩展代码（保留历史，不参与主应用开发）
├── resources/              # 静态资源文件
├── package.json            # 根package.json，统一管理脚本
└── README.md               # 项目说明文档
```

### 目录设计原则
1. **单一入口原则**：所有源代码统一放在 `src/` 下，根目录仅保留配置、文档、资源
2. **职责清晰**：`src/client/`、`src/server/`、`src/shared/` 三层分离，耦合度最低
3. **向后兼容**：`src/extension/` 保留历史扩展代码，与主应用完全解耦
4. **易于构建**：统一入口便于CI/CD配置和构建脚本管理

## 6. 部署模式
### 开发模式
- 前端运行在本地3000端口
- 后端运行在本地3001端口
- VS Code扩展开发模式连接本地服务

### 生产模式
- 前端打包后可嵌入扩展、也可以部署为静态站点
- 后端可以打包为单文件可执行程序，零依赖部署
- 支持一键安装启动，不需要用户配置环境
