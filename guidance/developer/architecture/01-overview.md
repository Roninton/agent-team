# 01 - 整体架构概览

本文档从全局视角解释多 AI 协作平台的架构设计。

---

## 🏗️ 三层解耦架构

本项目采用经典的三层解耦架构，各层职责单一，独立部署：

```
┌─────────────────────────────────────────┐
│          前端展示层 (Client)              │
│  React + Ant Design + Vite               │
│  3 个页面：Chat / Group / Settings       │
└─────────────────────┬───────────────────┘
                      │ REST API
                      │ WebSocket
                      ▼
┌─────────────────────────────────────────┐
│         后端服务层 (Server)               │
│  NestJS + TypeScript + TypeORM           │
│  6 个核心模块 + WebSocket 网关            │
└─────────────────────┬───────────────────┘
                      │ ACP 协议
                      │ 子进程 IPC
                      ▼
┌─────────────────────────────────────────┐
│      ACP 代理进程层 (External)            │
│  每个代理独立 Node.js 子进程              │
│  完全隔离，通过 ACP 协议通信              │
└─────────────────────────────────────────┘
```

## 📦 模块依赖关系

### 后端 6 个核心模块

```
                    WebSocket
                       │
                       ▼
┌─────────────────────────────────────────────┐
│            SessionService                    │ ← 会话状态管理
└──────┬──────────────────┬──────────────────┘
       │                  │
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│ AgentService│    │ MessageService│ ← 消息路由存储
└──────┬──────┘    └─────────────┘
       │
       ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ GroupService│    │ TaskService │    │ ContextService│
└─────────────┘    └─────────────┘    └─────────────┘
```

### 模块职责边界

| 模块 | 核心职责 | 不负责 |
|------|---------|--------|
| **Agent** | 代理进程生命周期管理、启动/停止、限流 | 不管理会话状态 |
| **Session** | 会话创建/关闭、状态追踪、WebSocket 推送 | 不管理消息内容 |
| **Message** | 消息存储、查询、索引、@提及解析 | 不管理代理进程 |
| **Group** | 群组 CRUD、成员管理 | 不处理消息路由 |
| **Task** | 任务状态机、代理/群组关联 | 不执行实际任务 |
| **Context** | 上下文存储、版本同步 | 不处理业务逻辑 |

---

## 🔄 数据流基本原则

1. **前端 → 后端**: 只能通过 REST API 或 WebSocket 通信
2. **模块间**: 只能通过 Service 层调用，禁止直接访问数据库
3. **后端 → 代理**: 只能通过 ACP 协议，禁止直接操作代理内存
4. **共享层**: 只能放纯类型定义和无副作用的工具函数

## 🎯 架构设计决策回顾

### 为什么用三层解耦？
- ✅ 前端可以独立更换（Web → VS Code 插件 → IDEA 插件）
- ✅ 后端可以独立扩展（单机 → 分布式集群）
- ✅ 代理层可以独立升级（ACP 协议版本更新不影响主平台）

### 为什么用 6 个独立模块而不是 1 个大模块？
- ✅ 单一职责原则：每个模块只做一件事
- ✅ 可测试性：每个模块可以独立单元测试
- ✅ 可维护性：修改一个模块不影响其他模块
- ✅ 瓶颈定位：性能问题可以精确到具体模块

### 为什么每个代理是独立子进程？
- ✅ 故障隔离：一个代理崩溃不影响整个平台
- ✅ 资源隔离：可以独立限制每个代理的 CPU/内存
- ✅ 安全隔离：代理代码运行在独立沙箱
- ✅ 协议解耦：只要符合 ACP 协议，用什么语言写代理都行

## 📊 性能瓶颈概览

按可能性从高到低排序：

| 层级 | 可能瓶颈 | 表现 | 分析方法 |
|------|---------|------|---------|
| 代理层 | 单个代理响应慢 | 某代理回复总是慢 | 见 06-performance-analysis.md |
| 后端 | 消息查询慢 | 聊天记录加载卡顿 | 数据库索引优化 |
| 后端 | WebSocket 推送频率高 | 大量消息时 UI 卡顿 | 批量推送、节流 |
| 前端 | 大列表重渲染 | 消息超过 1000 条后卡顿 | 虚拟列表、Memo 优化 |
| 前端 | Ant Design 组件重渲染 | 操作响应延迟 | 性能 Profiler 分析 |
