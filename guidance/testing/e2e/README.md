# E2E 测试指南

本指南用于指导 End-to-End 测试的编写与执行，覆盖单元测试难以覆盖的集成场景。

---

## E2E 测试 vs 单元测试

| 维度 | 单元测试 | E2E 测试 |
|------|---------|----------|
| **测试目标** | 单个函数/类/模块正确性 | 完整用户场景正确性 |
| **测试范围** | 单一模块 | 多模块协作 + 前后端交互 |
| **执行速度** | 快（ms级） | 慢（s级） |
| **维护成本** | 低 | 高 |
| **主要问题** | 集成bug测不到 | 定位问题困难 |

### 单元测试覆盖边界
✅ 能测：单一函数输入输出、单模块内部逻辑
❌ 难测：
- 完整用户操作流程（前端点击 → API → 进程 → WebSocket → 前端显示）
- 多代理并发场景
- 异常恢复（进程崩溃、网络断开）
- 性能边界（限流、连接上限）

**以上正是 E2E 测试的核心价值！**

## E2E 测试金字塔

```
        ┌────────────────────┐
        │   边界场景测试      │ ← 10~20个用例
        │  (异常、压力、边界)  │
        └────────────────────┘
    ┌────────────────────────────┐
    │      集成测试               │ ← 30~50个用例
    │  (核心场景完整流程)          │
    └────────────────────────────┘
┌────────────────────────────────────┐
│          冒烟测试                   │ ← 10个以内用例
│  (验证系统基本可用，能启动能操作)    │
└────────────────────────────────────┘
```

---

## 快速开始

### 1. 先写冒烟测试（优先级最高）
目标：确保每次构建后系统基本可用
- 服务能正常启动
- 前端页面能正常加载
- 核心功能点一下能用

### 2. 再写核心场景集成测试
目标：覆盖 5~10 个最常用的完整用户流程

### 3. 最后补充边界场景
目标：验证系统鲁棒性，异常场景能优雅处理

---

## E2E 测试编写原则

1. **一个用例只测一个完整场景**
   - ❌ 不要：测试启动代理的同时也测试发消息
   - ✅ 应该：分成两个独立用例

2. **用例之间完全独立**
   - 每个用例从零开始
   - 不依赖前一个用例的执行结果

3. **模拟真实用户行为**
   - 用真实的点击、输入、等待
   - 不要 mock 太多，否则失去 E2E 意义

4. **优先覆盖单元测试盲区**
   - 不要重复单元测试已经覆盖的逻辑
   - 重点测多模块、跨层、异常恢复
