# LFT 任务跟踪工作流（Lightweight Feature Tracking）

## 什么是LFT工作流
轻量级的任务驱动开发流程，所有开发任务都通过todo文件跟踪，确保每个改动都有明确的目的、验收标准和可追溯性，配合SDD规范实现文档驱动+任务驱动的双重保障。

## 目录结构
```
task/
├── todo/           # 待开发任务
├── in-progress/    # 开发中任务（同一时间只能有一个任务在此目录）
├── done/           # 已完成任务
├── archived/       # 已归档历史任务
└── README.md       # 本文件
```

## Todo 文件命名规范
采用 `YYYYMMDD-序号-简短任务标题.md` 格式，例如：
```
20260506-001-实现多代理群组管理功能.md
20260506-002-重构SessionManager支持多活跃会话.md
```

## Todo 文件内容模板
```markdown
# 任务标题

## 描述
清晰说明任务的目的、要解决的问题、实现的功能点。

## 关联规范
对应的SDD规范文档路径，例如：`docs/specs/features/multi-agent-group.md`

## 验收标准
1. [ ] 功能点1可以正常工作
2. [ ] 功能点2满足设计要求
3. [ ] 相关单元测试通过
4. [ ] 没有引入新的bug

## 优先级
P0 / P1 / P2 / P3

## 预计工时
X小时

## 相关人员
开发者：
评审者：

## 变更记录
- 2026-05-06 创建任务
```

## 工作流步骤

### 🔴 重要原则：大改动必须先建Task
当改动涉及以下情况时，**必须**先建立Task文件再开始开发：
1. 涉及多个文件的重构或功能开发
2. 需要查阅多个SDD规范文档
3. 预计工时超过2小时
4. 影响多个模块或子系统

### 大改动开发三部曲
1. **第一步：建立Task文件**
   - 在`todo/`目录创建Task文件
   - 填写任务描述、验收标准、优先级
   - 明确关联的SDD规范文档

2. **第二步：学习上下文并填入Task**
   - 按SPEC_INDEX.md的顺序阅读相关SDD文档
   - 将关键约束、设计原则、接口定义摘录到Task的「上下文信息」章节
   - 明确列出需要遵循的所有规范要求
   - 确保开发前对需求和约束有完整理解

3. **第三步：逐步执行并更新Todo**
   - 将任务拆解为具体的执行步骤（分阶段）
   - 每完成一个步骤立即在Task中标记✅
   - 遇到问题或变更及时更新Task记录
   - 完成后进行最终验证

### 常规开发流程
1. **新建任务**：在`todo/`目录下创建todo文件，填写完整信息
2. **开始开发**：将要开发的任务移动到`in-progress/`目录（同一时间只能有一个任务在这里）
3. **完成开发**：任务开发完成、验证通过后，移动到`done/`目录
4. **定期归档**：每个版本发布后，将`done/`目录中已上线的任务移动到`archived/`目录按版本归档
