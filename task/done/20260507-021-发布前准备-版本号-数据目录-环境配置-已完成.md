### 配置分层设计（三级配置）

**优先级从高到低：**
```
环境变量 > 配置文件 (config.yml) > 代码默认值
```

#### 配置文件搜索顺序
启动时按以下顺序查找配置文件，找到即停止：
1. 命令行参数指定：`--config /path/to/config.yml`
2. 环境变量指定：`ACP_CONFIG_FILE`
3. 用户配置目录：`~/.config/acp-platform/config.yml`（Linux/Mac）
4. 项目内配置：`./data/config/config.yml`（默认位置）
5. 如都找不到，使用代码默认值

#### 目录结构设计原则

```
acp-platform/
├── config/              # 📁 配置文件目录（独立，可挂载）
│   └── config.yml       # 用户自定义配置
│
├── data/                # 📁 运行时数据目录（动态生成，需要备份）
│   ├── db/              # 数据库文件
│   ├── logs/            # 日志文件
│   └── agents/          # 代理工作目录
│
└── src/                 # 源代码（只读）
```

**配置文件搜索顺序（优先级从高到低）：**
1. 命令行参数：`--config /path/to/config.yml`
2. 环境变量：`ACP_CONFIG_FILE`
3. 系统配置目录：`/etc/acp-platform/config.yml`（Linux）
4. 用户配置目录：`~/.config/acp-platform/config.yml`
5. 项目配置目录：`./config/config.yml` ✅ **推荐位置**
6. 如都找不到，使用代码默认值

#### 配置项清单

| 配置项 | YAML 路径 | 环境变量 | 默认值 | 说明 |
|--------|-----------|---------|--------|------|
| 服务端口 | `server.port` | `ACP_SERVER_PORT` | 3001 | 后端 HTTP 端口 |
| 主机绑定 | `server.host` | `ACP_SERVER_HOST` | '0.0.0.0' | 监听地址 |
| CORS 源 | `server.corsOrigin` | `ACP_CORS_ORIGIN` | '*' | 允许的前端源 |
| 数据库路径 | `database.path` | `ACP_DB_PATH` | `./data/db/acp.sqlite` | SQLite 数据库文件 |
| 日志目录 | `log.path` | `ACP_LOG_PATH` | `./data/logs/` | 日志文件目录 |
| 日志级别 | `log.level` | `ACP_LOG_LEVEL` | 'info' | debug/info/warn/error |
| 数据根目录 | `data.root` | `ACP_DATA_ROOT` | `./data/` | 运行时数据根目录 |
| 配置目录 | `config.path` | `ACP_CONFIG_PATH` | `./config/` | 配置文件目录 |
| 代理工作目录 | `agent.workDir` | `ACP_AGENT_WORKDIR` | `{data.root}/agents/` | 代理进程工作目录 |
| 最大代理数 | `agent.maxInstances` | `ACP_MAX_AGENTS` | 10 | 同时运行的代理上限 |
| 环境名称 | | `NODE_ENV` | 'development' | development/production |

#### config.yml 示例
```yaml
# ACP Platform 配置文件
server:
  port: 3001
  host: 0.0.0.0
  corsOrigin: "http://localhost:5173"

database:
  path: ./data/db/acp.sqlite

log:
  path: ./data/logs
  level: info

agent:
  maxInstances: 10
  workDir: ./data/agents

# 高级配置
advanced:
  # 是否在启动时自动创建缺失的目录
  autoCreateDirs: true
  # 是否打印完整配置（注意不要打印敏感信息）
  printConfigOnStartup: false
```

## � 配置读取位置与时机

### 在哪里读取配置？（最佳实践）

```
src/server/src/config/
├── config.module.ts       # NestJS 配置模块
├── config.service.ts      # 配置服务，统一读取入口
├── config.loader.ts       # 配置加载逻辑（YAML + 环境变量）
└── config.default.ts      # 默认值定义
```

**加载时机：**
1. NestJS 启动时，**最先**加载配置（在其他模块之前）
2. 先找配置文件，再合并环境变量
3. 验证所有必要配置是否合法（比如端口是数字）
4. 自动创建目录（如果配置了 autoCreateDirs）
5. 打印启动日志（包含版本号和数据目录位置）

### 为什么在这个位置？
- ✅ **单一入口**：所有模块通过 `ConfigService` 读取配置，不用自己解析
- ✅ **提前验证**：启动时就发现配置错误，不用等到运行时崩
- ✅ **可测试**：单元测试时可以注入 mock 配置
- ✅ **热重载**（可选）：后续可以支持配置修改不重启

### 各模块如何使用配置？
```typescript
// ✅ 正确：通过 ConfigService 读取
@Injectable()
export class AgentService {
  constructor(private configService: ConfigService) {}
  
  someMethod() {
    const maxAgents = this.configService.get('agent.maxInstances');
    // ...
  }
}

// ❌ 错误：不要到处读环境变量
process.env.ACP_MAX_AGENTS // 散落在各个文件，难以维护
```
